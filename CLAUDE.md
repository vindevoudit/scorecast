When making changes, go to your plans folder and check for Tier X.md where X is the tier to be prepared. You will find a detailed plan there to execute. Full-stack football prediction web app. React 18 + Vite frontend, Node/Express backend, PostgreSQL via Sequelize. Users make picks on games, join groups, send friend requests, comment + react on games, earn badges, and compete on probability-weighted leaderboards. Admins manage games and users from an in-app panel.

**Live**: https://bantryx.com (Azure Container Apps + Cloudflare DNS + Azure managed TLS). CD auto-deploys on push to `main` via [.github/workflows/deploy.yml](.github/workflows/deploy.yml).

## Tech stack at a glance

- **Frontend**: React 18 + Vite + Tailwind CSS
- **Backend**: Node 18+ / Express 4
- **DB**: PostgreSQL via Sequelize 6; migrations via sequelize-cli + umzug
- **Auth**: HttpOnly cookies — 15-min access JWT + 30-day rotating refresh (bcryptjs for passwords). Opt-in TOTP 2FA via `speakeasy` + `qrcode` with bcrypt-hashed recovery codes.
- **CSRF**: double-submit cookie (`sc_csrf` cookie + `X-CSRF-Token` header) on state-changing routes
- **CORS**: env allowlist via `CORS_ORIGINS`; throws on boot in production if unset
- **Security headers**: helmet (CSP, HSTS, X-Frame-Options: DENY, etc.)
- **Email**: pluggable transport via [lib/email.js](lib/email.js) — Resend when `RESEND_API_KEY` set; log-only fallback otherwise. Used by verify-email + password-reset flows.
- **Validation**: zod on every POST/PUT body
- **Rate limiting**: express-rate-limit on login, register, client-errors, comments, friend-requests, picks, forgot-password
- **Logging**: pino + pino-http with request-id correlation
- **Error reporting**: React `ErrorBoundary` + window listeners → `POST /api/client-errors` → structured log; Sentry opt-in via `SENTRY_DSN` / `VITE_SENTRY_DSN`
- **HTTP**: gzip via `compression` middleware
- **Leaderboard cache**: in-memory `Map` with 30s TTL
- **State (Tier 13.6/13.7)**: React Context — `AuthContext` (user + auth flow), `DataContext` (games/picks/groups/leaderboard/friends/profile + mutations), `NotificationContext` (toast banner). Selector hooks `useGames`/`usePicks`/`useGroups`/`useLeaderboard`/`useFriends` + `useRequest` (CSRF + 401 refresh-retry). App.jsx is a layout shell consuming hooks
- **Server layering (Tier 13.1–13.5)**: `routes/*.js` (parse + auth + service call) mounted at `/api`, `services/*.js` (domain logic: Pick / Game / Group / User / Comment / Leaderboard / Notification / Badge), `lib/` (cross-cutting infra: scoring, users, groups, friends, auth cookies/tokens, errors, response, errorMiddleware, leaderboardCache, email). `server.js` is composition shell only (≤ 200 lines)
- **Lint + format (Tier 9.1)**: ESLint 9 flat config + Prettier 3 + `husky` v9 + `lint-staged`. Pre-commit runs `lint-staged`; pre-push runs `npm run build`
- **Code-splitting (Tier 9.2)**: `React.lazy` + `<Suspense>` around AdminPanel, ProfileView, PicksHistory. Vite `manualChunks` splits `react`/`react-dom` (vendor) and `@sentry/*` (sentry) chunks. Build emits hidden sourcemaps for Sentry release upload

## Critical considerations (don't break these)

Every item below is a load-bearing invariant or gotcha that **isn't obvious from reading the code**. These are the things future-you will get burned by.

- **Scoring formula is duplicated**: [server.js](server.js) (`scorePick`, authoritative — used for leaderboard) and [src/utils/scoring.js](src/utils/scoring.js) (client-side preview). Must stay in sync in the same commit.
- **Design tokens, not color literals (Tier 11)**: every component must consume tokenized utilities (`bg-base`, `bg-elevated`, `text-fg`, `text-accent`, `border-default`, etc.) defined in [src/index.css](src/index.css) `:root` / `:root[data-theme='light']`. Hardcoded `slate-*` / `cyan-*` / `text-white` color classes are forbidden in `src/components/**` — they bypass the light/dark theme switch and look broken in the inverse theme. The theme system is binary (Light/Dark); `system` mode was removed in Chunk 3 — defaults to Dark (matches the brand). Legacy `'system'` values in `localStorage.sc_theme` normalize to `'dark'` on read.
- **Modal stacking + sidebar drawer (Tier 11)**: ConfirmModal + SignInModal + ProfileDrawer all `z-50`; toast viewport `z-[100]`; sidebar mobile drawer + NotificationBell dropdown `z-40`; OnboardingTour uses the Dialog primitive so `z-50` too. When a modal opens on top of the mobile drawer, the drawer's Escape handler is guarded by `drawerRef.contains(document.activeElement)` so Escape closes the modal first; the drawer stays open until focus returns into it. **Don't add a global `keydown` Escape listener that closes the drawer unconditionally** — it will steal Escape from any modal stacked above.
- **Onboarding tour gates (Tier 11 Chunk 4)**: `<OnboardingTour />` mounts when `user` is set, `!browseAsGuest`, `user.onboardingCompletedAt == null`, `view === 'games'`, and `games.length > 0`. New seed users (created via UI registration) trigger it; seed users in [tests/e2e/fixtures/seed.js](tests/e2e/fixtures/seed.js) ship with `onboardingCompletedAt: now` so the tour doesn't block existing E2E flows. The dedicated `tests/e2e/helpers/auth.js` `dismissOnboardingTour()` helper handles the runtime-registered case in `pick-and-result.spec.js`. Skip + Done both `POST /api/me/onboarding-completed`; the route is idempotent (preserves the existing timestamp on repeat calls).
- **Badge + notification side effects**: any code path that sets a result, creates a pick, creates a group, or accepts an invite must call `BadgeService.evaluateBadges()` and `NotificationService.notify()` (Tier 13.4 — moved out of server.js). Existing wires fire inside `services/{Pick,Game,Group}Service.js`: pick creation, result setting (single + bulk), group creation, invite-accept, friend-request/accept, public-group-join.
- **Route ordering**: `/api/groups/discover` is registered _before_ `/api/groups/:groupId` so Express doesn't treat `discover` as a path param. Same convention for any new `/api/groups/<literal>` route.
- **Reaction emoji palette is fixed**: 👍 ❤️ 😂 😮 🔥 — defined as `ALLOWED_EMOJIS` in [validation/schemas.js](validation/schemas.js) and `REACTION_EMOJIS` in [src/components/CommentThread.jsx](src/components/CommentThread.jsx). Adding an emoji requires editing both.
- **`pickMap` shape**: in [App.jsx](src/App.jsx) it stores full pick objects keyed by `gameId` (not just `choice`), so GameCard can pass `existingPickId` to the undo handler. Audit [GameCard.jsx](src/components/GameCard.jsx) (`existingChoice` / `existingPickId`) if you change the shape.
- **Avatars are deterministic**: [Avatar.jsx](src/components/Avatar.jsx) hashes the _lowercased username_ via FNV-1a → HSL. `displayName` never affects color — so renaming doesn't shuffle existing users' avatars.
- **Bulk admin self-skip**: the bulk-user endpoints filter the caller's own id and return it in `skipped: [{id, reason: 'self'}]` rather than erroring the whole batch.
- **`save({hooks: false})`** is intentional in the role-update endpoint, `PUT /api/me`, the bcrypt backfill seeder, and bulk role flips — without it, the `beforeUpdate` hook would re-hash an already-hashed password.
- **Migrations framework (Tier 5.1)**: **never** add raw DDL to `runMigrations()` — it's a thin umzug shim. Add a new file under [migrations/](migrations/) via `npx sequelize-cli migration:generate --name foo` and use `IF NOT EXISTS` guards. `migrations/` and `seeders/` are versioned source code — always commit them. See [MIGRATION_GUIDE.md](MIGRATION_GUIDE.md).
- **Leaderboard cache (Tier 5.2 / Tier 13.4)**: `GET /api/leaderboard` reads through [services/LeaderboardService.js](services/LeaderboardService.js) which wraps [lib/leaderboardCache.js](lib/leaderboardCache.js) (30s TTL). **Any new mutation that affects standings must call `LeaderboardService.invalidate('all'|'group:<id>')` from inside the owning service** — existing call sites: `PickService.{createPick,deletePick}`, `GameService.{setResult,bulkSetResult,bulkDelete,deleteGame}`, `GroupService.{acceptInvite,joinPublic,leave,deleteGroup}`, `UserService.{deleteUserById,bulkAction}` (on delete). Forgetting this means stale standings for up to 30 s.
- **Cascade transactions (Tier 5.3 / Tier 13.4)**: `UserService.cascadeDelete`, `GameService.cascadeDelete`, and `GroupService.cascadeDelete` accept a `{transaction}` option and forward it to every internal `destroy()`. Callers wrap with `await sequelize.transaction(async (t) => { await Service.cascadeDelete(x, {transaction: t}); })`. Bulk endpoints run **one transaction per entity** (a single bad row doesn't undo the rest). **`NotificationService.notify()` calls fire outside the transaction** so a rollback never produces ghost messages — keep that ordering.
- **Structured logging (Tier 5.4)**: use `req.log.error({err}, 'msg')` in handlers, top-level `logger.*` for boot-time messages. **Don't use `console.*`** — there should be zero such calls in backend code. The `X-Request-Id` response header is echoed back to clients; pass it through to correlate a client error with a server log line.
- **Frontend error reporting (Tier 5.4b)**: render errors bubble to [src/components/ErrorBoundary.jsx](src/components/ErrorBoundary.jsx); window-level errors + unhandled promise rejections go through [src/lib/clientErrorReporter.js](src/lib/clientErrorReporter.js) to `POST /api/client-errors`. Both paths also fire a `scorecast:client-error` DOM event that App.jsx listens for to show a transient toast. The boundary's raw error text is gated on `import.meta.env.DEV` — **don't surface it in prod**. Sentry is opt-in: server-side requires [lib/instrument.js](lib/instrument.js) to be the very first `require()` in [server.js](server.js) (OpenTelemetry instrumentation); browser-side is a dynamic import gated on `VITE_SENTRY_DSN` so Vite tree-shakes it when unset.
- **Helper return shapes (Tier 5.7)**: `getGroupsForUser()` returns `[{id, name, ownerId, visibility, members: [{userId, username}], invites: [{username, createdAt}], createdAt}]`; `getGroupById()` returns the same shape minus `visibility` (existing inconsistency, intentional). Preserve the shape — several components consume it.
- **Cookie auth (Tier 6.8)**: three cookies — `sc_access` (HttpOnly, 15-min JWT, Path=/), `sc_refresh` (HttpOnly, 30-day opaque token, **Path=/api/auth** so it doesn't go out on every request), `sc_csrf` (readable, 30-day). **Bearer-header auth is gone** — `authMiddleware` reads `req.cookies.sc_access` only. Login/register respond `{user}` (no token in body). Refresh tokens are **rotating** — each `POST /api/auth/refresh` revokes the inbound row and issues a new pair. Frontend `request()` ([src/App.jsx](src/App.jsx)) catches a 401, tries refresh once, retries the original; `/api/auth/*` calls are exempt from the retry to prevent recursion.
- **CSRF (Tier 6.7)**: [middleware/csrf.js](middleware/csrf.js) double-submit pattern. Every state-changing request (POST/PUT/PATCH/DELETE) must send `X-CSRF-Token: <sc_csrf cookie>`. Frontend `request()` reads via [src/lib/cookies.js](src/lib/cookies.js) and adds the header automatically. **Exempt routes** (in `EXEMPT_PATHS`): `/api/login`, `/api/register`, `/api/auth/refresh`, `/api/auth/verify-email`, `/api/auth/forgot-password`, `/api/auth/reset-password`, `/api/client-errors`. Add new pre-auth or anonymous mutating endpoints to that set; otherwise rely on the default CSRF check.
- **Account lockout (Tier 6.6)**: 5 failed logins → 15-min lock (`users.lockedUntil`). Counter resets on success. Response is identical 401 "Invalid credentials" for wrong-pw / unknown-user / locked — no enumeration. Lockout state is also **cleared on password reset**.
- **Token storage pattern (Tier 6.4/6.5/6.8)**: high-entropy tokens (verify-email, password-reset, refresh) are stored as **SHA-256 hashes** indexed by `tokenHash` — O(1) lookup, no bcrypt. The raw token only exists in transit (email URL or cookie value). **Recovery codes are different** — they're human-readable (low entropy) so they're **bcrypt-hashed** at rounds 8, looped through on verify. Don't mix the two patterns.
- **Email service (Tier 6.3)**: [lib/email.js](lib/email.js) `send({to, subject, html, text})` **never throws** — a failed transport must not break registration or password reset. When `RESEND_API_KEY` is unset, it logs the payload to stdout in dev (handy for grabbing verify/reset links from server logs). Wire new outbound emails through this module, not raw transport calls.
- **2FA (Tier 6.9)**: opt-in TOTP via `speakeasy` + `qrcode`. **Setup is two-step**: `POST /api/me/2fa/setup` stores an _unconfirmed_ `totpSecret` + 10 bcrypt-hashed recovery codes, returns raw codes ONCE. User must then `POST /api/me/2fa/confirm` with a valid code to set `totpEnabledAt`. Login with `totpEnabledAt` set issues `sc_challenge` cookie (HttpOnly, 5-min JWT, Path=/api/auth) and returns `{challenge: true}` **instead** of auth cookies — auth cookies are only issued by `POST /api/auth/2fa/verify` after a valid code or recovery code. Used recovery codes are **spliced out** of the array; no regenerate-without-disable endpoint.
- **Password reset cascading (Tier 6.4 + 6.8)**: `POST /api/auth/reset-password` does three things atomically — updates the password (Sequelize `beforeUpdate` re-hashes), clears lockout state, and revokes **all** refresh tokens for the user (`revokeAllUserRefreshTokens`). - **CD pipeline image tags (Tier 9.7)**: every push to `main` builds an image tagged with `${{ github.sha }}` AND `latest`. The migration Job is updated to the new image **before** the app, so migrations always apply before the new revision serves traffic. If a migration fails, the `migrate` job exits non-zero and CD halts — no traffic shift. To recover: revert the bad migration and re-push.
- **Cascade-delete fix-up (post-Tier 11)**: when prod was first deployed, `models/index.js` ran `sequelize.sync({ alter: false })` BEFORE `runMigrations()`. Sync created the user-owned FKs (token tables, notifications, badges, etc.) with the default `NO ACTION` because the Sequelize associations didn't declare `onDelete`. The shipped `CREATE TABLE IF NOT EXISTS … ON DELETE CASCADE` migrations then no-op'd against the already-synced tables, so prod's FKs were stuck on NO ACTION and `UserService.cascadeDelete` 500'd whenever the target had any verify-email / reset / refresh / notification / badge row (an earlier note in this file incorrectly claimed "production isn't affected" — it was). Three-part fix: (1) `cascadeDelete` now destroys those rows explicitly inside the transaction ([services/UserService.js:20](services/UserService.js#L20)); (2) every `User.hasMany(...)` association now declares `onDelete: 'CASCADE'` so fresh `sync()` deploys are correct ([models/index.js:47](models/index.js#L47)); (3) [migrations/20260516000002-cascade-user-fks.js](migrations/20260516000002-cascade-user-fks.js) `DROP CONSTRAINT IF EXISTS` + re-`ADD CONSTRAINT … ON DELETE CASCADE` retrofits the running prod DB. The admin-panel spec used to wipe `refresh_tokens` rows before the cascade test to dodge this bug — that workaround is now unnecessary but left in place because it's harmless. **Don't call `closeDb()` in a spec's `afterAll`** — `workers: 1` means the `require('models')` Sequelize pool is shared across specs; closing it stalls every later spec.
- **Profile privacy (Tier 8.6)**: every user has `users.profileVisibility` — `'public'` (default; existing behavior), `'friends'` (only accepted friends see the full profile), `'private'` (only self + admins). The gate lives in [services/UserService.js:getProfileByUsername](services/UserService.js) and produces an identical 404 for both `friends`-gated-out and `private` so the friend graph isn't probeable through response codes. Leaderboards still include non-public users by rank + points but mask the `username` (replaced with `displayName` if set, else `Player #<first 4 hex of uuid>`); masking is applied by `LeaderboardService.getOverallForViewer` / `getForGroupForViewer` on top of the viewer-agnostic 30-s cache. **Group implicit social contract**: within a per-group leaderboard, members never see each other masked regardless of their visibility (`exemptIds` = the group's own membership). **Admin override**: admins always see unmasked rows + full profiles. **Friend requests bypass the gate** — `/api/users/search` returns every match with a `profileVisibility` flag (username stays in the response since friend requests need it; the client may render the row masked). **Cache invalidation**: `PUT /api/me` invalidates the `'all'` leaderboard cache when `displayName` or `profileVisibility` actually changes — without that, the masking layer would project against stale visibility for up to 30 s. **Anonymous viewers** of a public group's `/api/groups/:id` get a per-member-masked list (`GroupService.maskMembersForAnon`).
- **Frontend login error race (uncovered by Tier 5.5b)**: `AuthContext.handleLogin` calls `showStatus('Invalid credentials')` then re-throws the error. `AuthView.handleLogin` doesn't catch the re-throw, so it surfaces as an unhandled promise rejection → `clientErrorReporter` fires the `scorecast:client-error` event → `NotificationContext` clobbers the status banner with "Something went wrong — refresh if things look off." The user briefly sees the right message, then the wrong one. UI-level lockout assertions should match the API response (status code / `error` field) directly rather than the status banner. Fix is to catch in AuthView (or stop re-throwing in AuthContext on user-facing errors) — left as a separate cleanup, not part of Tier 5.5b.
- **Football API status mapping (Tier 4b)**: upstream → local status lives in **one place** — [lib/fixtureStatus.js](lib/fixtureStatus.js) `STATUS_MAP` + `mapUpstreamStatus()` + `deriveResultFromFixture()`. Both `LeagueService.upsertFixture` (manual + daily sync) and `GameService.applyLiveUpdate` (60-s live-score job) import from here so they can never drift. `SUSPENDED` maps to `'in-progress'` (not `'cancelled'`) because suspended matches usually resume; `EXTRA_TIME` + `PENALTY_SHOOTOUT` map to `'in-progress'`. Result derivation prefers `score.winner` (HOME_TEAM / AWAY_TEAM / DRAW) over score comparison so penalty-shootout knockouts (1-1 fullTime but a defined winner) resolve correctly. Draws stay `result=null` with `status='finished'` — the existing `result` enum is `'home' | 'away'` only.
- **Live-score reconcile (Tier 4b Chunk 2)**: the live-score job polls `?status=LIVE,IN_PLAY,PAUSED` — matches that finished _between ticks_ drop off that list and would otherwise stay `status='in-progress'` forever locally. The job's reconcile pass at the bottom of [lib/jobs/syncLiveScores.js](lib/jobs/syncLiveScores.js) finds local in-progress rows whose `sourceId` wasn't in the LIVE response and batch-fetches via `footballApi.getMatchesByIds()` to write the final state. **Don't remove the reconcile** — without it, every finished match becomes stale forever until the next daily sync.
- **Live-minute estimate is client-side (Tier 4b Chunk 2)**: the football-data.org free tier doesn't expose `minute` / `injuryTime`. [src/utils/time.js](src/utils/time.js) `matchMinute(kickoff, {halfTimeReached, phase})` computes from kickoff + refines with two persisted signals: `games.halfTimeReached` (true once upstream wrote `score.halfTime`) and `games.phase` (`regular` / `extra-time` / `penalty-shootout` from `score.duration`). Display rules: `phase='extra-time'` → `"ET"`; `phase='penalty-shootout'` → `"PEN"`; halfTime reached + elapsed in 46-60 → `"HT"`; halfTime NOT reached + elapsed > 45 → `"45'"` (don't claim 2nd-half minutes). Post-HT the displayed minute is shifted down by 15 to compensate for the wall-clock break. **`halfTimeReached` is monotonic** in `applyLiveUpdate` — never flips back to false on an upstream blip.
- **Game status ↔ result sync (Tier 4b Chunk 2)**: `GameService.setResult` + `bulkSetResult` flip `status` alongside `result` (set → `'finished'`, clear → `'scheduled'`). Without this, manual admin results would set `result` but leave `status='scheduled'` — and the frontend `useGames` bucketing (which now consults `status` first) would misclassify them. Frontend selectors should preserve "status first, result fallback" ordering so legacy hand-entered rows still classify correctly.
- **Scheduler advisory locks (Tier 4b Chunk 2)**: [lib/scheduler.js](lib/scheduler.js) wraps every job in `pg_try_advisory_lock(crc32(jobName))` so a multi-replica deploy (post Tier 10.4) only runs any given tick once. `crc32` is deterministic and stable across deploys — **don't switch to a random or per-process lock id** or the multi-replica safety goes away. Jobs failing inside the lock are logged + skipped; the lock always releases via `finally`. node-cron is in-process so a missed tick is silently the next tick's problem; fixture sync is idempotent next day, live-score self-recovers on the next minute. `scheduler.start()` is a no-op when `NODE_ENV=test` so Playwright doesn't spawn surprise jobs.
- **Cron skipped without API key (Tier 4b Chunk 2)**: both jobs early-return when `FOOTBALL_DATA_API_KEY` is unset (avoids a 503-loop every minute in dev without a key). Logs at `info` for syncLiveScores and `warn` for syncFixtures — local dev without a key sees nothing alarming.
- **football-data.org status filter syntax (Tier 4b Chunk 2)**: the v4 API rejects repeated `?status=X&status=Y` (returns 400) and wants **comma-separated** `?status=X,Y,Z`. [lib/footballApi.js](lib/footballApi.js) `getLiveMatches()` uses the comma form; don't switch back to repeated params.
- **`games.leagueId` is NOT NULL (Tier 4b Chunk 3)**: enforced by [migration 20260518000007-games-tighten-league-not-null.js](migrations/20260518000007-games-tighten-league-not-null.js). Any pre-Tier-4b orphan games are auto-backfilled into a synthetic `Legacy / Imported` league (`sourceProvider='legacy'`, `sourceLeagueId='LEGACY'`, `active=false`) + a current-year season. **New games must always have a `leagueId`** — admin-created games through `POST /api/admin/games` should accept (and the schema may need to require) a league. Today's admin form doesn't, but the legacy league catches it. The migration is idempotent — safe to re-run.
- **Audit log payload truncation (Tier 4b Chunk 3)**: [services/AuditLogService.js](services/AuditLogService.js) `truncatePayload()` replaces >4KB serialized payloads with `{_truncated: true, _bytes, preview: '...first 512 chars...'}`. The middleware fire-and-forgets via `res.on('finish')` and **never throws back into the request lifecycle** — an audit-log outage can never block the real mutation. Failed-auth attempts (401/403 thrown before `auditMutation` runs) are NOT audited — `authMiddleware` runs first; if you want auth-failure audits you'd need to wire `auditMutation` earlier in the stack.
- **League picker URL contract (Tier 4b Chunk 3)**: [src/components/GameFiltersBar.jsx](src/components/GameFiltersBar.jsx) reads `?league=PL&season=2026` once on mount (after the leagues fetch lands) and writes via `history.replaceState`. The URL uses the **football-data.org `sourceLeagueId` code** (`PL`, `BSA`, etc.) — not our internal UUID — so links are shareable + stable. The picker resolves code → UUID against `/api/leagues` before calling the filtered `GET /api/games?leagueId=<uuid>`. Both filters are stored on `DataContext.gameFilters` so `refreshGames` (called after picks, admin mutations, etc.) preserves the active filter.
- **ML pipeline lives in [ml/](ml/), is a separate Python project**: not part of the Node app. Don't add Python deps to `package.json`; don't try to bake the ML container into the Node Dockerfile. The pipeline is a CONSUMER of `games.homeProbability` / `games.awayProbability` columns (writes through the admin API, audit-logged). Schema additions: none. Full deep-dive in [ARCHITECTURE.md §8.17](ARCHITECTURE.md) + [ml/ONBOARDING.md](ml/ONBOARDING.md).
- **ML username regex gotcha**: the service-account user is **`ml_pipeline`** (underscore) NOT `ml-pipeline` (hyphen) — `registerSchema` in [validation/schemas.js:11](validation/schemas.js#L11) enforces `^[A-Za-z0-9_]+$`. Stale docs that say `ml-pipeline` are wrong; fix on sight.
- **ML walk-forward invariant**: features for any match must be built from data strictly dated BEFORE that match. Elo's `home_elo_pre` / `away_elo_pre` columns + form's `compute_form(team_history, as_of, last_n)` filter on `date < as_of` enforce this. **Don't take shortcuts** that compute form against the full match list as-of-today — that's the canonical data leak that gives flattering log-loss with no out-of-sample value.
- **ML probability write invariant**: writes go through `PUT /api/admin/games/:id`. The 3-class XGBoost output is collapsed to 2-class via draw redistribution (`home_out = P_h / (P_h + P_a)`), rounded to `DECIMAL(3,2)`, re-balanced so the pair sums to exactly 1.00 (larger side keeps its rounded value), and nudged off the `(0.50, 0.50)` "untouched by anyone" sentinel from `LeagueService.upsertFixture`. Skipping any of these breaks either the `updateGameSchema` validator or the next run's idempotency.
- **HFA default = 0**: [ml/scorecast_ml/elo/engine.py](ml/scorecast_ml/elo/engine.py) `EloConfig.home_field_advantage` defaults to 0, not the conventional 65. Ablation in [ml/scripts/compare_hfa.py](ml/scripts/compare_hfa.py) showed it's a structural no-op for XGBoost (trees absorb the constant shift). The home/away feature pair structure carries the actual home-advantage signal. Pass `--hfa 65` if you genuinely want to reproduce the legacy training.
- **Promoted-team rating**: `EloConfig.promoted_team_strategy = "min_rating"` (default). After the first season, any team appearing for the first time enters at `min(current ratings)` — captures the empirical reality that promoted sides underperform the bottom of the league they're joining. Set to `"initial"` for vanilla Elo if needed.
- **`load_latest_bundle` strict match**: only matches `{league}_YYYY-MM-DD.joblib`, NOT suffixed variants. A/B-test artifacts (`PL_2025-05-25_hfa0.joblib`, `_hfa65.joblib`, `_5season_uncal.joblib`) are deliberately ignored so they don't accidentally become production. Load by explicit path via `load_bundle(path)` when comparing models.
- **Isotonic calibration fits on val, evaluated on test (Phase 2)**: [ml/scorecast_ml/train/model.py](ml/scorecast_ml/train/model.py) `fit_calibrators(bundle, X_val, y_val)` fits one `IsotonicRegression` per class against raw val proba, attaches to `bundle.calibrators`. `predict_proba` then applies the calibrators + renormalizes rows to sum to 1.0 (isotonic per-class doesn't preserve the simplex). **Val mlogloss reported AFTER calibration is by-design optimistic** — calibration fit on the same data it's measured on. Honest OOS check is the held-out test set (currently `scripts/backtest_2526.py` against the in-progress DB season). Bundles trained pre-Phase-2 don't have a `calibrators` attribute — `getattr(self, 'calibrators', None)` fallback in `predict_proba` handles them gracefully.
