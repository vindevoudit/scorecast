Full-stack football prediction web app. React 18 + Vite frontend, Node/Express backend, PostgreSQL via Sequelize. Users make picks on games, join groups, send friend requests, comment + react on games, earn badges, and compete on probability-weighted leaderboards. Admins manage games and users from an in-app panel.

**Live**: https://bantryx.com (Azure Container Apps + Cloudflare DNS + Azure managed TLS). CD auto-deploys on push to `main` via [.github/workflows/deploy.yml](.github/workflows/deploy.yml).

## Tech stack at a glance

- **Frontend**: React 18 + Vite + Tailwind CSS
- **Backend**: Node 18+ / Express 4
- **DB**: PostgreSQL via Sequelize 6; migrations via sequelize-cli + umzug
- **Auth**: HttpOnly cookies — 15-min access JWT + 30-day rotating refresh (bcryptjs for passwords). ~~Opt-in TOTP 2FA via `speakeasy` + `qrcode` with bcrypt-hashed recovery codes.~~ 2FA was parked in Tier 22 for the marketing launch — see Critical considerations + routes/auth.js header for the revival recipe.
- **CSRF**: double-submit cookie (`sc_csrf` cookie + `X-CSRF-Token` header) on state-changing routes
- **CORS**: env allowlist via `CORS_ORIGINS`; throws on boot in production if unset. Non-prod fallback is a localhost-only allowlist (Tier 22 M2 — was previously `true`).
- **Security headers**: helmet (CSP, HSTS with `preload`, X-Frame-Options: DENY, etc.) + extended Permissions-Policy (camera/mic/geo/payment/USB/fullscreen/motion-sensors/interest-cohort all denied) (Tier 22 M3 + L4)
- **Email**: pluggable transport via [lib/email.js](lib/email.js) — Resend when `RESEND_API_KEY` set; log-only fallback otherwise. Used by verify-email + password-reset flows.
- **Validation**: zod on every POST/PUT body
- **Rate limiting**: express-rate-limit on login, register, client-errors, comments, friend-requests, picks, forgot-password, group password-join, /me/{password,email} (`sensitiveAccountLimiter`, 10/hr/IP — Tier 22 H1), /me/{push-preferences,onboarding-completed,accept-terms} + PUT /me + /notifications/{:id/read,read-all} (`lightWriteLimiter`, 60/min/IP — Tier 22 H1), group invite (`inviteLimiter`, 5/min/IP — Tier 22 H2)
- **Logging**: pino + pino-http with request-id correlation
- **Error reporting**: React `ErrorBoundary` + window listeners → `POST /api/client-errors` → structured log; Sentry opt-in via `SENTRY_DSN` / `VITE_SENTRY_DSN`
- **HTTP**: gzip via `compression` middleware
- **Leaderboard cache**: in-memory `Map` with 30s TTL atop materialized `user_scores` + `user_scores_overall` tables (Tier 24 — incremental dual-writer on every score-affecting mutation; reads are sub-ms indexed SELECTs regardless of user count)
- **State (Tier 13.6/13.7)**: React Context — `AuthContext` (user + auth flow), `DataContext` (games/picks/groups/leaderboard/friends/profile + mutations), `NotificationContext` (toast banner). Selector hooks `useGames`/`usePicks`/`useGroups`/`useLeaderboard`/`useFriends` + `useRequest` (CSRF + 401 refresh-retry). App.jsx is a layout shell consuming hooks

- **External football data (Tier 4b + Tier 18)**: [football-data.org v4](https://www.football-data.org/) TIER_ONE plan (20 req/min, paid since 2026-05-23 — was free 10 req/min) via [lib/footballApi.js](lib/footballApi.js) — provider-agnostic surface (`getCompetitions` / `getFixtures` / `getLiveMatches` / `getMatchesByIds`). Daily fixture sync (cron `0 3 * * *` UTC) + 30-s live-score poll (Tier 18, was 60-s) managed by [lib/scheduler.js](lib/scheduler.js) (node-cron + `pg_try_advisory_lock(crc32(jobName))` for multi-replica safety). Live-score job uses a single global `?status=LIVE,IN_PLAY,PAUSED` call regardless of how many leagues are tracked, followed by a reconcile pass via `?ids=` for any local in-progress games that fell off the LIVE filter (caught the IN_PLAY → FINISHED transition window). Picks remain winner-only (multi-kind deferred). Cost: €19/mo

Both are 2-feature `(home_elo, away_elo)` boosters consumed by [services/PredictionService.js](services/PredictionService.js) `MODEL_PATHS[sourceLeagueId]`. For neutral-venue fixtures (default for WC, manually settable via `games.neutralVenue`) the cascade symmetrizes inference by averaging `predict(home, away)` + `predict(away, home)` swapped — guarantees `predict(A, B) === predict(B, A)` mirror invariant. Schema additions in the games table: `games.neutralVenue BOOLEAN DEFAULT FALSE`, `games.eloKMultiplier NUMERIC(4,2) NULL` (null = 1.0).
- **Audit log (Tier 4b)**: [middleware/auditLog.js](middleware/auditLog.js) `auditMutation(action, entityType)` wraps every mutating `/api/admin/*` endpoint; records via `res.on('finish')` so the final status code reflects the real outcome. Payloads truncated at 4KB. Read-only paginated view at `GET /api/admin/audit-log` + the AuditLog admin tab

## Critical considerations (don't break these)

Every item below is a load-bearing invariant or gotcha that **isn't obvious from reading the code**. These are the things future-you will get burned by.

- **2FA is parked, not deleted (Tier 22 — commit `b2bd286`)**: the route handlers, frontend components, and zod schemas were removed; the `users.{totpSecret, totpEnabledAt, totpRecoveryCodes}` columns + every migration + the `lib/auth.js` `CHALLENGE_COOKIE`/`CHALLENGE_TTL_MS` constants + the `speakeasy`/`qrcode` npm deps are preserved by design. Revival: `git revert <commit-1-SHA>` restores everything end-to-end with zero schema work. **Before reviving**, audit `users.totpEnabledAt != null` — none should exist today (migration `20260514000001-disable-all-2fa.js` already cleared them all in May), but if any creep in, decide whether to enforce 2FA for those users immediately on the revival commit OR wipe the columns first. **The `twoFactorEnabled` field in GET /api/me stays in the response** as a no-op signal until revival — don't remove it. **The marker comment at the top of routes/auth.js + routes/me.js IS the revival recipe** — keep them intact.
- **Web Push endpoint allowlist (Tier 22 H4)**: [validation/schemas.js](validation/schemas.js) `pushSubscribeSchema.endpoint` refine() rejects any URL whose host isn't in `PUSH_ENDPOINT_HOSTS` (FCM / Apple / Mozilla) or matching `PUSH_ENDPOINT_HOST_SUFFIXES` (`.notify.windows.com`, `.push.apple.com`). Plus an HTTPS-only check. New push provider = schema bump in both places. [services/PushService.js](services/PushService.js) `sendToSubscription` carries a defensive private/loopback-IP block (127._, 10._, 172.16-31._, 192.168._, 169.254.\* incl. Azure IMDS, ::1, fc/fe80) that drops the subscription on send rather than risking an SSRF outbound — covers DNS-rebind + stale-row scenarios that bypass the schema gate at write time. **Don't bypass either layer** when adding push providers.
- **Group-comment edit/delete re-checks membership (Tier 22 H3 IDOR)**: [services/CommentService.js](services/CommentService.js) `edit()` and `remove()` call `assertStillMember(comment, userId)` for any comment with a `groupId`. Without this, a user who leaves a group can still PUT-rewrite (or DELETE) their old comments inside it, defeating the membership gate `create()` enforces. Admin override on `remove()` bypasses the membership check (admin scope outranks group membership); `edit()` has no admin override (admins shouldn't be impersonating other authors).
- **MAX_GROUP_MEMBERS cap (Tier 22 M4; raised to 2000 post-Tier-24)**: [services/GroupService.js](services/GroupService.js) caps every group at 2000 members (env-overridable via `MAX_GROUP_MEMBERS`, clamped to [10, 5000]). Enforced inside the 4 add-member paths: `joinPublic`, `joinWithPassword`, `acceptInvite`, `approveJoinRequest`.
